# 框架分析

这套框架整体上是合理的，而且适合现在这个阶段的实验。

它不是那种非常“工程产品化”的训练框架，而是更偏研究代码：  
重点是能快速改模型、改数据、改 loss、改训练策略，然后把实验跑起来。

对我们现在这件事来说，它是够用的。

## 1. 这套框架在做什么

现在这套流程可以简单理解成：

1. 用配置文件决定训练怎么跑  
2. 用 `Trainer` 串起模型、数据、loss、优化器、验证、保存 checkpoint  
3. 用 `MVInverse` 做材质预测  
4. 用 `Pi3X` 的一部分预训练权重给 `MVInverse` 做初始化  
5. 在新数据集上继续训练

也就是说，这个框架最核心的目标不是“做一个通用平台”，而是：

“方便做多视图材质重建相关的训练实验。”

## 2. 框架的几个核心部分

### 2.1 配置文件

配置文件在 `training/config/` 下面，比如：

- `textureverse_pi3x_docker.yaml`
- `textureverse_pi3x_docker_mvinverse_style.yaml`

配置文件负责决定：

- 用哪个模型
- 用哪个数据集
- 图像大小是多少
- 一次采多少张多视图图像
- 冻结哪些模块
- loss 怎么配
- checkpoint 怎么保存

这部分设计是好的，因为很多实验差异其实都只是改配置，不需要反复改 Python 代码。

### 2.2 Trainer

`training/trainer.py` 是训练框架的中心。

它主要负责：

- 初始化环境和分布式训练
- 创建模型、loss、优化器
- 加载预训练权重或恢复训练
- 创建 train / val dataloader
- 执行训练循环
- 执行验证循环
- 记录日志和 TensorBoard
- 保存 checkpoint

如果把整个项目比作一台机器，`Trainer` 就像总控制器。

### 2.3 模型

模型主体是 `MVInverse`。

它的任务不是只预测一个东西，而是同时预测多个材质相关结果，比如：

- albedo
- metallic
- roughness
- normal
- shading

所以它本质上是一个多任务模型。

### 2.4 Loss

`training/loss.py` 负责把各个分支的监督组合起来。

大致规律是：

- albedo / metallic / roughness / shading  
  用“回归损失 + 梯度损失”

- normal  
  用方向相关的损失，也就是负余弦相似度

这部分也比较合理，因为材质图不只是数值接近就够了，边缘和局部结构也很重要。

### 2.5 数据集和动态采样

这套框架有一个很明显的特点：

它不是传统那种“每个样本固定一张图”的训练方式。  
它更像是：

- 先选一个场景
- 再从这个场景里随机抽若干张图
- 把这些图一起送进模型

所以一个 batch 的常见结构是：

- `B`：这次有几个场景
- `N`：每个场景采了几张图

`images` 的常见形状就是：

`[B, N, C, H, W]`

这也是为什么你前面改 `img_nums` 之后，显存会变化很大。

## 3. 为什么 Pi3X -> MVInverse 这个思路是成立的

我们前面已经分析过权重匹配情况，结论很清楚：

- `encoder`：100% 匹配
- `decoder`：100% 匹配
- `register_token`：100% 匹配
- `res_encoder`：0%
- 各个材质 head：0%

所以现在真实发生的事情是：

- `MVInverse` 的主干部分拿到了 `Pi3X` 的预训练能力
- `res_encoder` 和材质输出头还是随机初始化

这说明当前训练策略应该是：

- 冻结已经有预训练的部分
- 放开随机初始化的部分去学习

所以“`encoder + decoder` 继承预训练，`res_encoder + heads` 重新学”这个方向是合理的。

它明显比全随机训练更靠谱。

## 4. 这个框架合理的地方

### 4.1 结构清楚

模型、loss、数据、优化器、checkpoint 都是分开的。  
你要改其中一块，不需要把整个框架推倒重来。

### 4.2 很适合做实验

比如你想试这些东西：

- 冻结哪些模块
- 换哪个 checkpoint
- 换数据集
- 调 `img_nums`
- 调 loss 权重

这套框架都比较容易下手。

### 4.3 对多任务预测比较自然

因为 `MVInverse` 本来就不是单头模型，  
而 `loss.py` 已经按多分支方式组织好了，所以扩展起来比较顺。

### 4.4 验证图保存很实用

验证阶段会直接把：

- 输入图
- GT
- 预测结果
- mask

保存出来。

这对材质任务特别重要，因为很多时候只看 loss 不够，必须看图才知道模型到底在学什么。

### 4.5 checkpoint 逻辑现在已经比较顺手

现在已经支持：

- `last`
- `best`
- 每隔若干 epoch 额外保存一次

这对实验管理已经够用了。

## 5. 这个框架目前不够完美的地方

虽然整体合理，但它也不是没有问题。

### 5.1 Trainer 有点太大

现在 `trainer.py` 里面同时负责：

- 训练循环
- 验证循环
- 保存验证图
- best checkpoint 判定
- TensorBoard 记录
- checkpoint 保存

这会导致一个问题：

后面功能越加越多，`Trainer` 会越来越重，不太好维护。

### 5.2 验证指标现在还是偏实验型

现在的 `best` 是按 `PSNR` 来选的。  
这能用，但它不一定永远是最适合材质任务的指标。

比如：

- 对 albedo，PSNR 可能还比较直观
- 对 normal，方向误差可能比 PSNR 更重要
- 对 metallic / roughness，主观质量和 PSNR 也不一定完全一致

所以现在这个 best 策略是“能用的版本”，但以后可能还值得继续改。

### 5.3 动态多视图采样很灵活，但也更容易出问题

它的好处是：

- 更符合多视图任务
- 训练信号更丰富

但代价是：

- 显存波动大
- 更容易 OOM
- dataloader 更复杂
- 不同 epoch 的训练负载可能不太一样

所以这套框架不是“傻瓜式稳定”，而是“灵活但需要小心调参数”。

### 5.4 研究代码味道比较重

这套代码整体能看出来是从研究项目发展出来的。

优点是：

- 改起来快
- 做实验方便

缺点是：

- 有些逻辑比较集中
- 有些模块职责不够干净
- 长期维护时会越来越依赖“熟悉项目的人”

## 6. 对我们现在的任务来说，这个框架够不够

如果目标是：

- 先把 TextureVerse 跑通
- 验证 `Pi3X` 初始化是否有帮助
- 看看材质图能不能学出来
- 做一些冻结策略和 loss 的实验

那这套框架是够的，而且是合适的。

我不会建议现在重写。

因为现在真正更重要的不是“框架优雅不优雅”，而是：

- 数据是不是对
- 视图采样是不是合理
- 显存能不能扛住
- 哪些模块该冻结
- loss 和验证结果是不是正常

这些比“代码结构再抽象一层”更重要。

## 7. 我对当前框架的整体评价

一句话总结：

这套框架作为研究训练框架是合理的，作为长期大型工程框架还可以继续整理。

更直白一点说：

- 现在拿它做实验，没问题
- 现在就重构，不划算
- 等训练流程稳定了，再考虑进一步整理结构，会更合适

## 8. 之后如果要继续优化，优先级可以这样看

### 必须先解决的

- 训练能稳定跑通
- 显存不要频繁 OOM
- 数据和 mask 逻辑正确
- checkpoint 和 best 保存正常

### 建议后面再做的

- 把 `Trainer` 再拆干净一点
- 把验证指标做成可配置
- 把 best metric 明确成单一规则
- 统一文件编码，避免中文注释乱码

### 现在不急着做的

- 过度工程化重构
- 把整个训练框架改成特别通用的平台

因为当前阶段最重要的还是先把实验结论跑出来。
